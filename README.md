# Logisync — AI-Powered Humanitarian Logistics Platform

## Both Mobile (IOS + Android) and Web application

> Predict shortages before they happen. Dispatch the right resources from the right warehouse in one tap.

Logisync is a full-stack humanitarian logistics system built for crisis-zone supply chain management. It combines real-time inventory tracking, role-based dispatch workflows, a live geospatial map, and a proactive AI engine that detects demand surges hours before stock runs out — then automatically proposes an optimised resupply plan.

**Live deployment:**

## The Problem

Humanitarian logistics in active crisis zones fails not because of a lack of resources, but because of **information lag**. Warehouse workers discover shortages only after stock reaches zero. Dispatchers manually coordinate transfers across dozens of locations by phone. Medical supplies, fuel, and food sit idle in a surplus warehouse 80 km away while another location runs dry.

Logisync replaces that reactive model with a proactive, AI-assisted operations hub.

## Feature Overview

### 1. AI Demand Prediction & Auto-Rebalancing

The signature feature. A background prediction engine monitors demand patterns for every `(delivery point, resource)` pair and raises alerts before a shortage occurs.

**How it works:**

1. Demand readings are collected from sensors, workers, or direct entry
2. The engine computes a **Weighted Moving Average** over the last 14 readings — giving recent periods proportionally more weight
3. When the short-term trend diverges ≥ 20% above the long-term baseline, an alert is raised
4. A **confidence score** (0.5 – 1.0) is calculated and the **predicted time-to-stockout** is estimated
5. An **optimal rebalancing proposal** is generated: which warehouses to draw from, how many units, and estimated arrival time (via Haversine distance ÷ 60 km/h)
6. Dispatchers review the proposal and approve or reject it **in one click**

All of this runs automatically in the background every hour. For demos, a manual trigger endpoint is available: `POST /v1/ai/run`.

**Confidence formula:**
```
confidence = min(0.5 + (divergence − 0.20) / 0.40, 1.0)
```

| Divergence | Confidence | Severity |
|---|---|---|
| 20% | 0.50 | Elevated |
| 40% | 0.75 | High |
| 60%+ | 1.00 | Critical |

**Optional LLM rationale:** If a Groq API key is configured, each alert receives a two-sentence human-readable explanation generated by `llama-3.1-8b-instant`. Stored in the `rationale` column; gracefully absent if the call fails.

### 2. Live Geospatial Map

An interactive Leaflet map displays all 20 warehouses and 40 delivery points across Ukraine. Each pin is colour-coded:

- **Green (normal)** — stock is healthy
- **Amber (elevated)** — stock is below the safety threshold
- **Red (critical)** — active predictive alert or near-zero stock

Clicking a pin opens a detail panel with current inventory levels and active alerts. The map auto-refreshes every 30 seconds.

---

### 3. Offline-First

An Offline-First approach used in both Web and Mobile applications.

- **Status** - Whenever you have an internet connection, you would have a status online.
- **Offline** - If you are offline, your status would change, you can still move around all pages, but only cached data will be live.
- **Sync** - The second you are online, all changes will be synced and fetched.

---
### 4. Mobile App (Flutter)

A dedicated Flutter application for warehouse workers operating in the field — designed around the brief's explicit requirement for warehouse usability. Also, offline-first.

Key design constraints:
- **Min 48×48px tap targets** — works with gloves
- **16sp+ font size** — readable in poor warehouse lighting
- **High-contrast dark theme** — visible in direct sunlight
- **Bottom navigation** — thumb-reachable on large phones

**QR scanning workflow:** Every resource shelf has a printed QR label encoding `RESOURCE:{id}:POINT:{id}`. Worker taps Scan → camera decodes → Quick Action screen appears with current stock. Three actions: update demand, flag urgent, confirm delivery. Two taps from scan to action — no searching, no typing.

A demand update recorded on the phone triggers the AI analysis immediately in the backend. An alert can appear on the dispatcher's desktop within seconds of the scan — demonstrable in real time.

**Platform:** Android + iOS from a single Dart codebase.

---

### 5. Real-Time Inventory

Every warehouse tracks stock per resource across 30 resource types (food, medical, fuel, electronics, clothing, building materials). Inventory is searchable by name and filterable by category. Safe-stock thresholds are calculated automatically (20% of quantity) and highlighted visually.

Workers see only their own warehouse. Dispatchers and admins see all locations.

---

### 6. End-to-End Delivery Workflow

Full request lifecycle with status state machine:

```
pending → allocated → in_transit → delivered
                  ↘ cancelled
```

- **Workers** create delivery requests (single or multi-item)
- **Dispatchers** run allocation (`POST /v1/delivery-requests/allocate`), approve/reject individual allocations, and mark dispatches
- **Urgent** requests trigger immediate auto-allocation on creation
- Priority can be escalated at any point (`POST /v1/delivery-requests/:id/escalate`)

Dispatchers inherit all worker capabilities. Admins inherit all dispatcher and worker capabilities.

---

### 7. Allocation Management

The allocations page gives dispatchers a full view of all planned transfers with approve, reject, and dispatch actions. Each allocation shows the source warehouse name, resource name, quantity, and current status. Bulk approval is also available per-request.

---

### 8. Role-Based Access Control

Three roles with strict enforcement at both the API and UI layers:

| Role | Access |
|---|---|
| `worker` | Create delivery requests, view own inventory, record demand readings |
| `dispatcher` | All worker actions + approve/reject allocations, run allocation engine, view all inventory |
| `admin` | All dispatcher actions + manage users, view full audit log |

JWT tokens carry the role claim. UI hides inaccessible nav items and action buttons. API enforces roles independently (defence in depth).

---

### 9. Audit Log

Every delivery action (create, allocate, approve, dispatch, deliver, cancel) is written to an append-only audit table with actor ID, role, action type, entity, before/after values, IP address, and timestamp. Accessible to admins via the Admin panel with full-text search.

---


### 10. Dashboard

At-a-glance operations summary: total inventory value, active alert count, pending request count, recent allocation status. Data is pulled live from the same API endpoints.

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│                   Frontend (React 18 + Vite)             │
│                                                          │
│  Dashboard · Map · Inventory · Alerts · Delivery ·       │
│  Allocations · Admin (Audit Log) · Settings              │
│                                                          │
│  Feature-based structure: each feature owns its own      │
│  API layer, hooks, and components.                       │
└──────────────────────────┬──────────────────────────────┘
                           │ REST/JSON over HTTP
┌──────────────────────────▼──────────────────────────────┐
│              Backend API (Go 1.25 / Echo v5)             │
│                                                          │
│  Strict layered architecture — no DI framework:          │
│  Controller → Use Case → Repository → PostgreSQL         │
│                                                          │
│  ┌───────────────────────────────────────────────────┐  │
│  │              AI Prediction Engine                 │  │
│  │  WMA · Confidence · Shortfall · Rebalancing Score │  │
│  │  Background ticker (1h) + on-demand trigger       │  │
│  │  Optional: Groq LLM rationale per alert           │  │
│  └───────────────────────────────────────────────────┘  │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│                    PostgreSQL 16                          │
│                                                          │
│  warehouses · customers · users · resources              │
│  inventories · delivery_requests · allocations           │
│  demand_readings · predictive_alerts · proposals         │
│  rebalancing_transfers · audit_log                       │
└─────────────────────────────────────────────────────────┘
```

**Backend:** Go 1.25 · Echo v5 · pgx v5 · golang-migrate · swaggo  
**Frontend:** React 18 · TypeScript · Vite · Tailwind CSS · Lucide icons · Leaflet  
**Mobile:** Flutter 3 · Dart · iOS + Android · mobile_scanner · flutter_secure_storage  
**Infrastructure:** Docker Compose · AWS EC2 · GitHub Actions CI/CD  
**AI:** Weighted Moving Average · Haversine scoring · Optional Groq LLM (llama-3.1-8b-instant)

## CI/CD

Every push to `main` (touching `backend/**`) deploys automatically to EC2:

```
GitHub push → Actions: lint + build + test → SSH to EC2 → git pull → docker compose up --build
```

PR branches run lint, build, and test only — no deployment.

## Getting Started

### Prerequisites
- Docker + Docker Compose
- Go 1.25 (for local backend dev)
- Node 20 (for local frontend dev)

### Run with Docker

```bash
git clone https://github.com/Megidy/BestLviv2026Test.git logysinc
cd logysinc/backend

cp .env.example .env
# Set JWT_SECRET at minimum. Add GROQ_API_KEY to enable LLM rationale.

docker compose up -d
```

API: `http://localhost:8080`  
Swagger: `http://localhost:8080/swagger/index.html`

### Run locally (backend)

```bash
cd backend
make start-deps    # start PostgreSQL via docker compose
make migrate-up    # run all migrations + seed data
make start         # go run -race ./cmd/api
```

### Run locally (frontend)

```bash
cd frontend
npm install
npm run dev        # starts on http://localhost:5173
```

## Demo Walkthrough

1. **Login** as `dispatcher_w1` / `secret`
2. Open **Alerts** — 7 pre-seeded predictive alerts are visible (4 critical, 3 elevated)
3. Click any alert row to expand — see demand trend, time-to-shortfall, and the auto-generated rebalancing proposal with warehouse names and quantities
4. Click **Approve** — the proposal is marked approved, the alert resolves, and it disappears from the open queue
5. Open **Map** — red/amber pins show affected delivery points; click one to see its inventory and alerts
6. Open **Allocations** — see the transfer plan created by the approved proposal; click **Dispatch** to mark it in transit
7. Login as `admin_w1` — navigate to **Admin** to view the full audit trail of every action taken

To reset and re-trigger AI from a clean state:
```bash
# On the EC2 host:
cd ~/logysinc/backend
docker compose exec backend sh -c "migrate -path migrations/dev -database \$POSTGRES_CONNECTION_URI down && migrate up"
# Then: POST /v1/ai/run
```
